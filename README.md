# 17prtur — Нейросети в промышленном туризме

## Обзор проекта

**17prtur** — MVP веб-приложения для автоматической сборки промышленного тура под конкретную группу. Система помогает туроператору за 5-10 минут подобрать предприятия, транспорт, питание и размещение, сформировать программу по минутам, рассчитать стоимость и выдать клиенту публичную ссылку на бронирование.

Проект построен на простой архитектуре **клиент-сервер**:

- **Frontend**: нативный HTML/CSS/JavaScript без сборщика
- **Backend**: Node.js HTTP-сервер без внешних зависимостей
- **Данные**: JSON-файлы в `server/data`
- **LLM**: OpenAI-compatible `/v1/chat/completions` + fallback-алгоритм правил

## Функционал

### Для туроператора

- **Сборка нового тура** по параметрам группы:
  - город;
  - тип группы: школьники, студенты, бизнес-делегация;
  - количество человек;
  - длительность дневной программы: 3, 5 или 7 часов;
  - цель визита: профориентация, дегустация, обмен опытом, бизнес-туризм;
  - интересы группы;
  - точка сбора;
  - необходимость размещения.
- **Автоматический подбор объектов**:
  - 1-3 предприятия или культурно-промышленных объекта;
  - транспортная компания с подходящей вместимостью;
  - точка питания для туров от 5 часов;
  - размещение для сценария с ночёвкой.
- **Формирование программы тура**:
  - маршрут;
  - программа по минутам;
  - пояснение выбора;
  - стоимость на человека и на группу.
- **Ручная корректировка тура**:
  - замена предприятий;
  - замена транспорта;
  - замена питания;
  - замена размещения.
- **Публичная ссылка клиента** `/proposal/:code`.

### Для клиента

- **Просмотр готового предложения** по публичной ссылке.
- **Просмотр маршрута, программы и стоимости**.
- **Бронирование тура** через форму.
- **Эмуляция оплаты** для MVP.

### Для администратора

- **Просмотр справочников**:
  - предприятия;
  - объекты культуры;
  - питание;
  - транспорт;
  - размещение.
- **Настройка внутренней LLM**:
  - API URL;
  - модель;
  - API-ключ;
  - системный промпт.

## Роли пользователей и сценарии

| Роль | Описание | Основные сценарии |
|------|----------|-------------------|
| `operator` | Туроператор / менеджер турагентства | Собирает тур, редактирует маршрут, отправляет клиенту ссылку |
| `client` | Турист или представитель группы | Получает ссылку, смотрит предложение, бронирует и проходит mock-оплату |
| `admin` | Команда проекта | Настраивает LLM и ведёт справочники объектов |

### Основной сценарий MVP

1. Туроператор открывает главную страницу.
2. Вводит параметры группы.
3. Система фильтрует базу объектов по городу, вместимости, типу группы, времени работы и доступным стартам.
4. Если настроен API-ключ, backend вызывает LLM для ранжирования и пояснения выбора.
5. Если LLM недоступна, включается fallback-подбор по правилам.
6. Туроператор получает программу, маршрут и стоимость.
7. Туроператор вручную корректирует тур при необходимости.
8. Клиент получает публичную ссылку и бронирует тур.
9. Система создаёт mock-бронирование и mock-уведомление.

## API-маршруты

Базовый префикс: `/api`

### Каталог

- `GET /catalog` — получить предприятия, питание, размещение и транспорт

### Туры

- `GET /tours` — список созданных туров
- `POST /tours/plan` — собрать новый тур
- `PUT /tours/:id/selection` — вручную изменить выбор объектов и пересчитать тур

### Публичное предложение

- `GET /public/tours/:code` — получить тур по публичной ссылке
- `POST /public/tours/:code/book` — создать бронирование
- `POST /public/tours/:code/pay/mock` — выполнить mock-оплату

### Админка

- `GET /admin/ai-creds` — получить AI-настройки без раскрытия полного ключа
- `PUT /admin/ai-creds` — сохранить AI-настройки
- `POST /admin/ai-test` — проверить подключение к LLM
- `POST /admin/catalog/:collection` — добавить объект в справочник  
  Поддерживаемые коллекции: `enterprises`, `foodPlaces`, `accommodations`, `transportCompanies`

## LLM-интеграция

LLM-интеграция вынесена на backend, чтобы frontend не работал с API-ключами напрямую:

- API-ключ не передаётся на frontend;
- backend читает настройки из локального `server/data/ai-cred.json`;
- запрос отправляется в OpenAI-compatible endpoint `/v1/chat/completions`;
- модель получает параметры группы и уже отфильтрованных кандидатов;
- ответ модели используется для выбора и объяснения;
- если ключ не указан или провайдер недоступен, работает fallback-алгоритм.

### Безопасность ключей

`server/data/ai-cred.json` не хранится в репозитории и добавлен в `.gitignore`.

В GitHub отправляется только безопасный шаблон:

```text
server/data/ai-cred.example.json
```

Локальный файл настроек можно создать вручную или через UI:

```json
{
  "AI_API_URL": "https://api.openai.com/v1/chat/completions",
  "AI_API_KEY": "",
  "AI_API_MODEL": "gpt-3.5-turbo",
  "AI_SYSTEM_PROMPT": ""
}
```

### Логика fallback-подбора

Код проверяет жёсткие ограничения:

- город;
- тип группы;
- min/max размер группы;
- вместимость транспорта;
- часы работы;
- доступные времена старта;
- длительность программы;
- обязательное питание для туров от 5 часов;
- приоритет бесплатных предприятий для школьников.

LLM не должна считать финальную цену как единственный источник истины. Итоговая стоимость пересчитывается backend-кодом.

## Запуск и настройка

### Требования

- Node.js 20+
- Git, если нужно отправлять изменения в GitHub

### Локальный запуск

```bash
npm start
```

Приложение будет доступно по адресу:

```text
http://localhost:4173
```

### Запуск в режиме разработки

```bash
npm run dev
```

### Настройка LLM

1. Откройте приложение.
2. Перейдите в раздел **AI настройки**.
3. Укажите:
   - API URL или base URL;
   - модель;
   - API-ключ;
   - системный промпт.
4. Сохраните настройки.
5. Нажмите **Проверить подключение**.

Если ключ не указан, MVP продолжит работать на fallback-алгоритме без обращения к LLM.

После сборки тура в карточке результата отображается статус: **LLM подключилась** или **Fallback по правилам**. Если включился fallback, интерфейс показывает причину: нет ключа, ошибка API, невалидный JSON-ответ или другая проблема вызова.

Для OpenAI-compatible провайдеров можно указать как полный endpoint, так и base URL. Например, `https://api.deepseek.com` будет автоматически использован как `https://api.deepseek.com/chat/completions`.

### Проверка основного сценария

1. Открыть `http://localhost:4173`.
2. Заполнить параметры группы.
3. Нажать **Собрать тур**.
4. Проверить программу, маршрут и стоимость.
5. При необходимости изменить предприятия, питание или транспорт.
6. Открыть публичную ссылку клиента.
7. Создать бронирование и выполнить mock-оплату.
## Запуск в Docker

Проект запускается одним контейнером и публикует наружу только один порт.
Frontend уже является статикой (`public/`) и обслуживается тем же Node.js backend.
JSON-хранилище приложения находится внутри именованного Docker volume `app_data`.

```bash
cp .env.example .env
docker compose up --build -d
```

После запуска приложение доступно по адресу:

```text
http://localhost:4173
```

Проверка состояния:

```bash
docker compose ps
docker compose logs -f app
```

Остановка:

```bash
docker compose down
```

Для удаления сохранённых данных:

```bash
docker compose down -v
```

API-ключ не хранится в образе или репозитории. При необходимости укажите
`AI_API_KEY`, `AI_API_URL` и `AI_API_MODEL` в локальном `.env`.
